PROJECT I

LG Regen

November 2025 – December 2025

동남아 이슬람 국가 맞춤 ThinQ 활용 솔루션
동영상
산출문서

동남아(인도네시아·말레이시아) 무슬림 가정의 생활 패턴을 반영해 가전 제어 경험을 재구성한 스마트홈 솔루션입니다.
Flutter 앱–Spring Boot–MySQL 기반으로 기능을 구현하고, ESP32 + LCD 디바이스 연동까지 고려해 전체 데이터 흐름을 설계했습니다.

개발/아키텍처
  • Client: Flutter 기반 화면 구성 및 사용자 흐름 설계
  • Back-end: Spring Boot REST API로 요청 처리 및 비즈니스 로직 구성
  • DB: MySQL 저장/조회 중심 CRUD 구현
  • Build: Gradle 기반 의존성/빌드 관리로 실행 환경 일관화
  • Cloud: Naver Cloud(NCP) 기준 배포/운영 흐름 고려
  • Hardware: ESP32 + LCD 연동(앱 ↔ 서버 ↔ 디바이스) 데이터/명령 흐름 정리

데이터 처리는 Spring Boot 백엔드 + MySQL을 중심으로 구성했습니다.
Flutter 앱(클라이언트)에서 발생하는 요청을 REST API로 받아 서버에서 비즈니스 로직을 처리하고, 결과를 MySQL에 저장/조회하여 다시 앱에 응답하는 흐름으로 CRUD 기능을 구현했습니다.
빌드/의존성 관리는 Gradle 기반으로 진행해 실행 환경을 일관되게 유지했으며, 서비스는 Naver Cloud Platform(NCP) 환경을 기준으로 배포·운영 흐름을 고려해 구성했습니다.

또한 프로젝트는 앱만으로 끝나는 구조가 아니라 ESP32 + LCD 하드웨어와 연동되는 구조였기 때문에 “앱 ↔ 서버 ↔ 디바이스” 흐름을 끊김 없이 이어주는 데 집중했습니다.
앱에서 사용자가 기능을 실행하면 서버에서 요청을 처리하고, 디바이스 제어에 필요한 데이터/명령 흐름을 정리하여 하드웨어에서 상태가 표시되거나 동작 결과가 반영되도록 전체 연결 구조를 맞췄습니다.
이 과정에서 단순 CRUD를 넘어, Flutter – Spring Boot – MySQL – ESP32/LCD로 이어지는 “전체 데이터 흐름”을 기준으로 설계/구현 관점을 정리할 수 있었습니다.

담당 역할
  • 백엔드: REST API 설계 및 서버 비즈니스 로직 구현(Spring Boot)
  • DB: MySQL 기반 데이터 모델 설계 및 CRUD 흐름 구성
  • 하드웨어 설계: ESP32 + LCD 연동을 위한 데이터/명령 흐름 정리(앱 ↔ 서버 ↔ 디바이스)
  • 운영 관점: NCP 기준 배포/실행 흐름을 고려한 구조 정리
개발 영역
  • [전력 관리] 선불(토큰 추가/잔액·예상 사용기간 표시) · 후불(당월 예상 요금/당일·당월 사용량/청구 내역 조회) 흐름 설계

PROJECT II

Docker Container Monitoring System

November 2024 – February 2025

cAdvisor · Prometheus · Grafana 기반 컨테이너 모니터링
발표자료 논문(PDF)

Docker 환경에서 실행되는 컨테이너의 CPU · Memory · Network 사용량을 실시간으로 수집·저장·시각화하는 모니터링 시스템을 구현했습니다.
cAdvisor → Prometheus → Grafana 파이프라인을 Docker Compose로 구성해, 실행 환경을 일관되게 유지하고 대시보드에서 변화 추이를 한눈에 확인할 수 있도록 했습니다.

개발/아키텍처
  • cAdvisor: 컨테이너 리소스 메트릭 수집(컨테이너 단위 CPU/메모리/네트워크)
  • Prometheus: cAdvisor 엔드포인트를 주기적으로 스크래핑하여 시계열 데이터 저장
  • Grafana: Prometheus 데이터를 기반으로 대시보드 구성 및 실시간 시각화
  • Docker Compose: 모니터링 스택을 하나의 구성으로 묶어 배포/실행 자동화

시스템은 컨테이너(리소스) → cAdvisor 수집 → Prometheus 저장 → Grafana 시각화 흐름으로 구성했습니다. 특히 다수의 컨테이너를 동시에 실행하는 상황에서도, 각 컨테이너별 리소스 지표가 대시보드에서 분리되어 확인되도록 구성해 운영 관점에서 “병목 지점”을 빠르게 파악할 수 있게 했습니다.

담당 역할
  • 모니터링 스택 설계: cAdvisor/Prometheus/Grafana 역할 정의 및 데이터 흐름(수집→저장→시각화) 설계
  • 구성/배포: Docker Compose 기반 서비스 구성 및 실행 환경 표준화
  • Prometheus 설정: cAdvisor 스크래핑 타깃/주기 등 수집 설정 구성
  • Grafana 대시보드: CPU/Memory/Network 지표를 중심으로 실시간 모니터링 화면 구성
  • 테스트 시나리오 구성: 단일/다중 컨테이너 구동 및 부하 상황에서 지표 변화 확인
개발 영역
  • [컨테이너 지표 수집] cAdvisor 기반 컨테이너별 CPU/메모리/네트워크 메트릭 수집
  • [시계열 저장] Prometheus 스크래핑으로 메트릭 누적 및 조회 기반 마련
  • [시각화] Grafana에서 컨테이너 단위 대시보드 구성(리소스 변화 추이 확인)
  • [부하 검증] CPU 과부하 / 메모리 사용량 증가 / 네트워크 트래픽 유발 컨테이너(task0~task2)로 동작 검증
  • [확장 관점] 운영 중 임계치 초과 알림 등 확장 방향(알림/자동 경고) 적용 가능 구조로 정리
